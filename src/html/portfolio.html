Adama Corporate Homepage

Horizontal and Vertical Navigation
adama.com/en/brochure/

This was one of the first projects I worked on as an FE dev at Building Blocks, I worked with a couple other FE devs to build the front end for this site, my main tasks were the site furniture (header, footer, nav) and the corporate homepage.

It's the corporate homepage I want to talk about here as it was particularly complex, it's essentially 3 horizontal, scrollable sections and a 4th section at the bottom which is a standard vertically scrolling area.

Each horizontal section has a large full height banner and then a number of smaller areas to the right (or left in the RTL version), so almost like a carousel that operates both horizontally and vertically.

Horizontal Scrolling - This can take place by 3 methods:

  • Native scrolling via the scroll bar or using a scrollpad on laptop
  • Left & right buttons, JS will calculate and scroll to the next/prev area.
  • Keyboard left & right buttons, JS will also calculate and scroll you to the next/prev area.

Vertical Scrolling - This can take place by 3 methods:

  • Scroll down with mouse, JS will hijack and scroll for you, calculating and then scrolling to the next section up/down.
  • Keyboard up & down button, the JS will calculate and then scroll to the next section up/down.
  • Nav bar on the side of the screen, either go up down or select a section from the 'pips', JS will calculate and then scroll to the relavant section.

This was a very challenging but rewarding piece of work. The layout works by using JS to calculate the widths of each area for each section, we then have a section width that we apply so that the areas can be positioned inside that container then.

Vertically all the sections are inside a container which is the full height of the screen and the sections are scrollable within that div then and for scrolling we just need to know heights and we can scroll to each one as needed.

One of the frustration points of this feature was scroll hijacking with a magic mouse (mac) as there is inertia for each scroll, potentially with some mice on pc's too. So you'd want to scroll the user to the next section on scroll but instead would end up moving 2 or 3 sections down. To get around this I added a timeout to counter the inertia so the scroll is essentially disabled for a moment whilst the JS scroll takes place. Another issue was with trackpads and magic mouse whereby you could scroll horizontal but because there might be some upwards or downwards scroll too the JS scroll event would trigger. To get around this I had to build in some level of tollerance to distinguish between the two.

Adama Local Contacts

Google Map
adama.com/australia/en/contact-us/map/map.html

Something I seem to do a lot of is creating maps, at least once per project recently, unfortunately my most recent work is either not yet live or is behind logins and geo-ip blocks (like gauloises.de) so I'll reference Adama again.

This was possibly the first map integration I'd made so I've refined things a little since then but the code is essentially the same. Usually when I've made a map for a client it's to be able to show multiple locations, usually with a 'popup' box to accompany it.

This particular map is a Google map (I've also built maps using Mapbox, very similar) and queries a backend webservice for a list of locations (JSON) which I then cycle through and add to the map. In this example I've used a map plugin to allowing 'clustering' of the pins for when they're very close by each other (marker clusterer) and I tend to use 'infobox' for the popup boxes as you then have complete control over the markup for the popup.

The markup for the popup comes from a mustache template (I prefer seperation of JS and html so I very rarely add markup to the JS and if I do it's only very minor, incedently I also dislike styling through JS, so I'd rather add a class and use css to open/close something as opposed to using jQuery show/hide) which was something new to me at the time of creating this map but is now pretty standard for me (when I need to add markup dynamically).

The map icons were interesting as I had to allow the to be content manageable (everything I build is for a CMS so I have to make everything very robust and allow for it to be modified) and also map them to a map icon type as there were 3-4 different kinds of pins/contact types.

To deal with this I simply used a script block on the page (the backend will update it) that extends the settings for this JS module, allowing the icons to be set, class names to be selected, etc. I then just had to check for these values in the JS and use them if they existed, thereby making it pretty adaptable.

So this map required the use of the google maps api (v3), the integration of marker-clusterer, infobox js and the use of Mustache templating. So all in this simple map required a fair bit of work in the end, a good learning curve though that has given me a good level of knowledge for future work on maps of various kinds and for all manor of requirements.
